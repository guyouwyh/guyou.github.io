# 引言

Spring 框架是一个分层架构，由 7 个定义良好的模块组成。Spring 模块构建在核心容器之上，核心容器定义了创建、配置和管理 bean 的方式 .

Spring 框架是一个集众多设计模式与一身的开源的、轻量级的项目管理框架，致力于 JAVAEE 轻量级解决方案。

# Spring 框架两大核心机制

  • IoC（控制反转）/ DI（依赖注入）
  • AOP（面向切面编程）

Spring 是一个企业级开发框架，是软件设计层面的框架，优势在于可以将应用程序进行分层，开发者可以自主选择组件。

# IOC

控制反转是一种通过描述（XML 或注解）并通过第三方去生产或获取特定对象的方式。在 Spring 中实现控制反转的是 IoC 容器，其实现方法是依赖注入（Dependency Injection,DI），依赖注入：就是利用 set 方法来进行注入的。

IOC: 将原有项目中手动通过 new 关键字创建对象权利交给 spring, 由工厂去创建对象，spring 框架不仅要创建对象还要在创建对象的同时维护类与类之间的依赖关系，因此在 IOC 的基础上引出了 DI 的概念。

# 依赖注入（Dependency Injection,DI）

概念

  • 依赖注入（Dependency Injection,DI）。

  • 依赖：指 Bean 对象的创建依赖于容器 . Bean 对象的依赖资源 .

  • 注入：指 Bean 对象所依赖的资源，由容器来设置和装配 .

  • 通过配置 bean 标签来完成对象的管理。

    • id ：对象名。
    • class ：对象的模版类（所有交给 IoC 容器来管理的类必须有无参构造函数，因为 Spring 底层是通过反射机制来创建对象，调用的是无参构造）
    • depend-on : 依赖创建，可以改变 bean 标签的创建顺序

  • 对象的成员变量通过 property 标签完成赋值。

    • name ：成员变量名。
    • value ：成员变量值（基本数据类型，String 可以直接赋值，如果是其他引用类型，不能通过 value 赋值）
    • ref ：将 IoC 中的另外一个 bean 赋给当前的成员变量（DI）

# Set 注入

通过 bean 标签在 Spring 容器中使用无参构造创建对象，spring 中通过实体类的 set 方法给对象中的属性赋值，id 是唯一的。

注意:

如果是 8 种基本类型 + String, 使用 value 进行注入，如果是日期类型，需要遵从格式 yyyy/MM/DD HH:mm:ss

如果是数组、list、map,set, 使用 <arrar> 、 list 、 map 、 set 标签

# 工厂模式

Spring 中工厂创建对象的模式

  • 默认 Spring 创建的是单例 (Singleton)

  • 静态工厂

工厂本身不需要创建对象，通过静态方法调用，对象 = 工厂名。工厂方法

factory-method ：执行的静态工厂方法的方法名

constructor-arg : 注入参数值

class : 静态工厂的全限定类名

  • 实例工厂

工厂本身需要创建对象。

通过工厂对象的方法创建需要的对象。

要先创建工厂，然后通过 factory-bean 创建的 bean 工厂 id, factory-method 是工厂中的方法名。也需要用 constructor-arg 给方法参数注入值.

可以实现 Spring 提供的接口 FactoryBean 创建工厂对象，只要实现这个接口，spring 都认为是一个工厂对象

  • ioc 容器创建时候不会创建工厂对象
  • 只有获取的时候工厂才会创建对象

还需要在配置文件中注册该 bean

autowire

参数:

  • default/no: 不进行自动装配
  • byName: 通过 id 名与属性名进行自动装配，如果找不到则赋值为 null
  • byType: 通过类型进行自动装配，如果存在多个相同类型，会抛出异常，如果是自动装配 List 集合，则会将多个类型都装配进 list 集合。

List 集合情况

  • Constructor: 根据构造器参数类型进行自动装配，如果存在多个类型，则根据参数名和 id 值进行匹配，如果找不到就会赋值为 null;

# 注解自动装配

Spring 的自动装配需要从两个角度来实现，或者说是两个操作：

  1. 组件扫描 (component scanning)：spring 会自动发现应用上下文中所创建的 bean；
  2. 自动装配 (autowiring)：spring 自动满足 bean 之间的依赖，也就是我们说的 IoC/DI；
  • @Autowire

首先是按类型自动装配，如果同时存在多个类型，则 id 与变量名相同的进行自动装配，如果找不到则报错

参数 require 默认为 true, 可以设置为 false, 这样找不到就不会报错，会赋值为 null

  • @ Qualifier

可以指定名称进行自动注入，如果找不到也会报错

自动装配执行是在类构造器之后执行。

  • @Resource

如有指定的 name 属性，先按该属性进行 byName 方式查找装配；其次再进行默认的 byName 方式进行装配；如果以上都不成功，则按 byType 的方式自动装配。都不成功，则报异常。

# Spring 四种常用注解

  • # @Controller : 一般用于声明控制器.

  • # @Service : 声明业务层

  • # @Reposity : 声明持久层

  • # @Component : 声明是 ioc 容器中的一个组件

四种注解都是为 IOC 容器中注册组件，在底层不区分，分成四个只是让我们在写代码的时候层次更清晰。

要在配置文件中进行包扫描，这样才能将有注解的类加入 ioc 容器中 (必须导入 aop 包)

注意：用注解加入 ioc 容器组件的 id 默认是类名首字母小写，注册的组件是单例的。

注解参数默认是 value, 可以给注册到 ioc 容器中的组件配置 id 值。

# scope 作用域

@Scoper 可以改变注册组件的模式

Spring 管理的 bean 是根据 scope 来生成的，表示 bean 的作用域，共 4 种，默认值是 singleton。

  • singleton：单例，表示通过 IoC 容器获取的 bean 是唯一的。
  • prototype：原型，表示通过 IoC 容器获取的 bean 是不同的。
  • request：请求，表示在一次 HTTP 请求内有效。
  • session：回话，表示在一个用户会话内有效。

request 和 session 只适用于 Web 项目，大多数情况下，使用单例和原型较多。

prototype 模式当业务代码获取 IoC 容器中的 bean 时，Spring 才去调用无参构造创建对应的 bean。

singleton 模式无论业务代码是否获取 IoC 容器中的 bean，Spring 容器在加载 spring.xml 时就会创建 bean。

# BeanFactory 和 ApplicationContext 的区别

  • ApplicationContext 是 BeanFactory 的子接口
  • BeanFactory:bean 工厂接口；负责创建 bean 实例，容器里面保存的所有单例 bean 其实是一个 map, 是 spring 中最底层的接口
  • ApplicationContext: 是容器接口；更多的负责容器功能的实现；
  • BeanFactory 最底层的接口，ApplicationContext 留给程序员使用的 ioc 容器接口，ApplicationContext 是 BeanFactory 的子接口

# AOP

AOP：Aspect Oriented Programming 面向切面编程。

AOP 是对面向对象编程的一个补充，在运行时，动态地将代码切入到类的指定方法、指定位置上的编程思想就是面向切面编程。将不同方法的同一个位置抽象成一个切面对象，对该切面对象进行编程就是 AOP。

AOP 的优点：

  • 降低模块之间的耦合度。
  • 使系统更容易扩展。
  • 更好的代码复用。
  • 非业务代码更加集中，不分散，便于统一管理。
  • 业务代码更加简洁存粹，不参杂其他代码的影响。

AOP 是对面向对象编程的一个补充，在运行时，动态地将代码切入到类的指定方法、指定位置上的编程思想就是面向切面编程。将不同方法的同一个位置抽象成一个切面对象，对该切面对象进行编程就是 AOP。

基本概念

  • 切面 (aspect): 横切关注点被模块化的抽象对象，通知和切入点的结合，通常是一个类
  • 通知 (advice): 想要加入的功能，例如日志，安全等
  • 连接点 (JoinPoint): 允许使用通知的地方，例如方法的前后
  • 切入点 (Pointcut): 根据自己的需求，符合自己需求的连接点。

# AOP 的总结

注解实现 aop 的步骤:

1. 使用 @Component 将切面类和实现功能增强的类加入到 ioc 容器中

2. 使用 @Aspect 注解指定哪个类是切面类

3. 使用注解声明通知类型

4. 在通知类型的 value 值里写切入点表达式

配置文件实现 aop 的步骤：

1. 使用 bean 标签将切面类和实现功能增强的类加入到 ioc 容器中

2. 使用 aop:config 标签指定哪个类的切面类

3. 使用 aop:before等 声明通知类型， method 属性是绑定要增强的方法， pointcut 是切入点表达式

注解和配置文件的合理使用

  • 重要的使用配置文件，不重要的使用注解
  • 注解快速方便，配置文件功能完善

# 事务

事务分为编程式事务处理和声明式事务处理

编程式事务处理 ：通过在业务层中注入事务管理对象，然后通过编码的方式进行事务控制

声明式事务处理 ：通过利用 aop 切面编程进行事务控制，并对事务属性在配置文件中完成细粒度配置

注册和配置事务管理器

事务管理器：相当于一个对事务进行操作的切面

使用配置文件进行事务配置

通过注解进行事务配置

事务细节

事务的传播 ：就在多个业务层之间相互调用时传递事务的过程称之为事务传播

Isolation : 事务的隔离级别

  • DEFAULT : 使用数据库的默认隔离级别

  • READ_UNCOMMITTED : 读未提交，一个用户读取到了另一个用户未提交的数据，脏读现象

  • READ_COMMITTED : 读已提交

  • REPEATABLE_READ : 可重复读，主要避免不可重复读现象出现

  • SERIALIZABLE : 序列化，主要为了避免幻读

注意：隔离级别越高，查询效率越低

propagation ：事务的传播途径，如果有多个事务嵌套运行，子事务是否和大事务共用一个事务。

  • ** REQUIRED 😗* 如果外层没有事务，则开启新的事务，如果外层存在事务，则融入当前事务。原理是将之前事务的 connection 传入当前事务
  • REQUIRED_NEW : 如果外层存在事务，外层事务挂起，自己开启新的事务执行，执行完成，回复外层事务继续执行
  • SUPPORTS : 支持事务，如果外层没有事务，不会开启新的事务，如果外层有事务，会融入事务
  • NOT_SUPPORTED : 不支持事务，如果外层存在事务，外层事务挂起，自己以非事务的方式执行，执行完毕，恢复外层事务
  • NEVER : 不允许外层有事务，存在事务就报错
  • MANDATORY : 强制外层事务
  • NESTED : 嵌套事务 (很少用)

noRollbackFor : 哪些事务可以不回滚，参数是一个 class 数组

RollbackFor : 哪些事务可以回滚，参数也是一个 class 数组，默认运行时异常会回滚

编译时异常不会回滚

readonly : 设置事务为只读，可以加快查询速度，只有查询才被允许，默认属性是 false

timeout : 事务超出指定时长后终止并自动回滚，单位为秒，设置 - 1 永不超时

如果事务传播途径是 REQUIRED , 事务的属性都是继承于大事务的，比如小事务调整超时时间是无用的。

本类方法的调用只有一个事务

# Spring 总结

Beanfactory 和 ApplicationContext 的区别

  • BeanFactory:bean 工厂的接口，是 spring 最底层的接口，负责创建 bean 的实例，容器里保存的所有单例 bean 其实是一个 map
  • ApplicationContext: 是容器接口，更多负责的是功能的实现。可以从 map 中获取 bean.
  • ApplicationContext 是 BeanFactory 的子接口

Spring 中最大的模式就是工厂模式，配置文件相当于一本说明书，告诉工厂该如何创建对象，工厂拿到说明书之后根据说明书创造对象。
